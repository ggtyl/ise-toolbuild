# ise-toolbuild

A search-based software engineering tool for improving single-elimination bracket fairness in Tournado.

This project applies search-based software engineering (SBSE) to improve the structural fairness of brackets generated by a software component in Tournado, a web-based tournament management system. The tool optimises the participant ordering given to a sequential single-elimination bracket generator, using Random Search and a Genetic Algorithm to minimise a fairness score based on standard seeding conventions.

## What the Tool Does

The baseline bracket generator produces brackets by assigning byes to the first participants in the input list and pairing the remaining participants sequentially. This approach is simple and deterministic, but it does not account for bracket fairness.

This tool treats participant ordering as a search problem. It evaluates candidate orderings using a structural fairness score and applies two optimisation methods:

- Random Search
- Genetic Algorithm with tournament selection, Order Crossover, swap mutation, and elitism

The goal is to find participant orderings that produce fairer brackets than the sequential baseline.

## Setup

This project was developed and tested with Python 3.12.

1. Create and activate a virtual environment:
   ```bash
   python3 -m venv venv
   source venv/bin/activate
   ```

2. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```

## Running

Run all commands from the repository root with the virtual environment activated.

Run the full experiment:
```bash
python3 src/experiment.py
```

Run statistical analysis:
```bash
python3 src/analysis.py
```

Generate figures and summary table:
```bash
python3 src/visualise.py
```

Run individual smoke tests:
```bash
python3 src/generator.py
python3 src/scoring.py
python3 src/random_search.py
python3 src/genetic_algorithm.py
```

## Experimental Setup Summary

The tool is evaluated across seven tournament sizes:

- 8
- 12
- 16
- 24
- 32
- 48
- 64 participants

For each tournament size:

- 30 independent runs are performed
- seeds 1 to 30 are used for reproducibility
- the evaluation budget is scaled with tournament size

Budgets used:

- 8: 100 evaluations
- 12: 200 evaluations
- 16: 300 evaluations
- 24: 500 evaluations
- 32: 750 evaluations
- 48: 1000 evaluations
- 64: 1500 evaluations

Statistical comparison is performed using the Wilcoxon signed-rank test.

## Results Summary

Across all evaluated tournament sizes, both Random Search and the Genetic Algorithm improve bracket fairness relative to the sequential baseline.

The Genetic Algorithm achieves lower mean fairness scores than Random Search for every tested tournament size. The difference is not statistically significant for 8 participants, but becomes statistically significant from 12 participants onwards.

Mean percentage improvement over the baseline:

| Participants | Random Search | Genetic Algorithm |
|---|---:|---:|
| 8  | 91.6% | 94.6% |
| 12 | 65.0% | 67.6% |
| 16 | 81.4% | 86.7% |
| 24 | 60.5% | 63.7% |
| 32 | 74.7% | 81.8% |
| 48 | 54.1% | 59.9% |
| 64 | 70.4% | 78.6% |

Raw results are stored in `results/results.csv`, and generated figures are stored in `results/`.

## Assumptions

The current implementation makes the following assumptions:

- participant `id` represents seed rank
- lower seed number means a stronger participant
- stronger participants always advance when simulating later rounds
- fairness is measured structurally using seeding-based rules, rather than from observed match outcomes

## Limitations

The optimiser only controls participant ordering. It does not directly control bye placement, because the underlying baseline generator always assigns byes to the first participants in the input list.

As a result, non-power-of-2 tournament sizes have a structural lower bound on the fairness score. This affects how much improvement can be achieved for sizes such as 12, 24, and 48 participants.
